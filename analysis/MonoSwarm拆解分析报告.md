# MonoSwarm 拆解分析报告

**报告级别**: CEO 决策参考  
**编制日期**: 2026-05-12  
**分析依据**: 源码审计（1,880行）+ 架构文档

## 1. 执行摘要

**一句话结论**: MonoSwarm 是一个架构清晰的原型级多Agent调度系统，核心机制（任务拆解、Wave并行、Context Passing）已跑通，但距离生产可用还差3个P0修复和1次架构升级。

| 维度 | 评分 | 核心判断 |
|------|------|----------|
| 技术可行性 | ✅ 已验证 | 端到端链路跑通，SQLite持久化稳定 |
| 任务拆解能力 | ❌ 不及格 | 纯关键词匹配，覆盖面<10%，漏匹配率极高 |
| 并发模型 | ⚠️ 伪异步 | subprocess同步等待，实际是串行+表面并行 |
| 系统可靠性 | ⚠️ 基础可用 | 无单元测试、无监控、foreign_keys关闭 |
| 扩展性 | ⚠️ 受限 | 2小时硬编码超时、无会话管理、无Worker数据共享 |

**必须立即修复的3件事**（预计1-2人天）:
1. 开启 SQLite `foreign_keys=ON`（1行代码，防范脏数据）
2. 修复 `subprocess shell=True` 注入风险（改为list传参）
3. 建立基础单元测试（至少覆盖Harness调度链路）

**3个月内必须完成的架构升级**:
- 引入LLM智能拆解，替代纯规则匹配
- 实现真正的异步执行引擎
- 增加可观测性（metrics + 监控仪表盘）

**风险评级**: 🟡 中等风险 — 核心链路可工作，但P0缺陷会在复杂场景下导致任务失败或数据损坏。

## 2. 现状快照

### 2.1 系统规模

| 指标 | 数值 | 说明 |
|------|------|------|
| 核心代码行数 | 1,880行 | ceo_brain(760) + harness(375) + worker_pool(307) + session_store(329) + run(101) + config(8) |
| 模块数量 | 6个 | CEO Brain、Harness、WorkerPool、SessionStore、Config、CLI |
| 数据库表 | 4张 | tasks、workers、event_log、ceo_assignments |
| Worker类型 | 6种 | code/research/doc/qa/ppt/ui |
| 并发上限 | 7个 | max_concurrent硬编码 |
| 超时设置 | 2小时 | worker_timeout硬编码 |

### 2.2 核心调用链路

```
用户输入 → CEOBrain.run_full_flow()
              ├─ decompose()       # 任务拆解（纯规则，约40关键词）
              ├─ execute()        # Wave分层执行
              │    └─ Harness.run() → WorkerPool.spawn() → subprocess.hermes chat
              └─ aggregate()      # Markdown报告输出
```

### 2.3 数据持久化现状

- **存储**: SQLite单文件（`~/.hermes/agent-swarm/swarm.db`）
- **优点**: 无服务器依赖，零运维；4张表结构完整；event_log只增不减
- **致命缺陷**: `foreign_keys=OFF`，级联删除无效，可产生脏数据

### 2.4 能力雷达图（自评）

```
任务拆解   ▓▓░░░░░░░░  20%   纯关键词，漏匹配率高
并发调度   ▓▓▓▓░░░░░░  40%   伪异步，subprocess阻塞
重试机制   ▓▓▓░░░░░░░  30%   基础3次重试，无差异化策略
持久化     ▓▓▓▓▓░░░░░  60%   SQLite完整但缺约束
会话管理   ▓░░░░░░░░░  10%   session_id复用worker_id
可观测性   ▓░░░░░░░░░  10%   无metrics，无监控
```

## 3. 核心价值评估

### 3.1 架构亮点

**✅ 极简依赖**: 仅Python标准库 + 外部Hermès CLI，无第三方包，学习成本低，部署简单。

**✅ 模块边界清晰**: CEO（决策）→ Harness（调度）→ WorkerPool（执行）→ SessionStore（持久化），职责分工明确，易于定位问题。

**✅ Context Passing设计合理**: 通过workspace目录 + context dict传递上下游信息，Worker间解耦。

**✅ Wave级并行**: 基于依赖拓扑排序，同层无依赖Worker强制并行，不同层顺序执行。

**✅ Skill动态路由**: Worker类型 → goal内容 → skill自动映射，已接入`software-development/skill-creator`、`research/arxiv`等。

### 3.2 核心价值定位

MonoSwarm 的本质是**多Agent任务调度基础设施**，而非完整的AI Agent产品。它的价值在于：

1. **任务自动化**: 把复杂任务拆解为可并行的子任务，减少人工协调成本
2. **产出可追溯**: SQLite持久化 + event_log，任务生命周期完整记录
3. **扩展预留**: `_llm_decompose()`已写好待启用，架构支持平滑升级

## 4. 不足矩阵（按优先级排列）

### P0 — 阻断性问题（必须立即修复）

| # | 不足 | 影响 | 修复成本 |
|---|------|------|----------|
| P0-1 | **任务拆解漏匹配率高** | 约90%场景无法正确识别Worker类型，用户体验灾难 | 1-2人天（短期关键词扩充）或 2-4周（LLM拆解） |
| P0-2 | **subprocess shell=True 注入风险** | 安全漏洞，若goal含特殊字符可导致命令注入 | 2小时（改为list传参） |
| P0-3 | **完全无单元测试** | 任何代码修改都是盲改，回归风险极高 | 1-2人天（建立基础测试） |

### P1 — 影响效率问题（1个月内修复）

| # | 不足 | 影响 | 修复成本 |
|---|------|------|----------|
| P1-1 | **SQLite foreign_keys=OFF** | 级联删除无效，删除task后workers残留，产生脏数据 | 1行代码 |
| P1-2 | **伪异步执行** | subprocess同步等待，实际串行，表面并发 | 2-3人天（重构为真正的异步） |
| P1-3 | **无可观测性** | 故障靠人工查日志，定位慢 | 1周（增加metrics表） |
| P1-4 | **重试无差异化策略** | 所有失败同一重试逻辑，治标不治本 | 3-5人天 |
| P1-5 | **2小时超时硬编码** | 无法适配不同任务类型，短任务浪费，长任务中断 | 1人天（配置化） |

### P2 — 体验优化问题（1-2个月内优化）

| # | 不足 | 影响 | 修复成本 |
|---|------|------|----------|
| P2-1 | **无真正会话管理** | session_id复用worker_id，跨Worker无法共享上下文 | 2-3人天（重构session_store） |
| P2-2 | **结果仅文本汇总** | 无结构化数据，无法做二次分析 | 1周（结构化输出） |
| P2-3 | **parent_id未启用** | 无任务层级树，无法表达复杂依赖 | 2人天（启用字段） |
| P2-4 | **代码305行偏长** | 单文件职责过重，维护困难 | 1周（模块拆分） |

### 4.1 漏匹配场景实测

| 用户任务 | 期望Worker | 实际结果 | 原因 |
|---------|-----------|---------|------|
| "帮我调试这个API为什么报错" | code_worker | generic_worker | "调试"未收录 |
| "调研一下竞品的功能差异" | research_worker | generic_worker | "调研"未收录 |
| "帮我优化下这个SQL查询性能" | code_worker | generic_worker | "优化"/"SQL"未收录 |
| "做一个用户增长的数据分析" | research_worker | generic_worker | "分析"/"增长"未收录 |
| "把这份PDF转成Word" | doc_worker | generic_worker | "转"/"Word"未收录 |

**根因**: 关键词词库规模约40词，覆盖面不到10%，且无同义词映射和模糊匹配。

## 7. 风险提示

### 7.1 技术风险

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| LLM拆解延迟不稳定 | 中 | 高 | 规则引擎作为兜底，超时强制回退 |
| SQLite单文件瓶颈 | 低 | 中 | 评估PostgreSQL迁移（Phase 3） |
| Worker进程泄露 | 中 | 中 | 增加process监控和超时强制kill |
| hermes chat CLI变更 | 低 | 高 | 封装CLI调用，解耦依赖 |

### 7.2 产品风险

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| 拆解质量不达预期 | 中 | 高 | 人工评估集持续迭代，设定准确率目标 |
| 用户期望过高 | 高 | 中 | 明确系统定位为"辅助工具"而非"全自动Agent" |
| 复杂任务依赖失控 | 中 | 中 | Phase 3实现任务树可视化，支持人工干预 |

### 7.3 业务连续性风险

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| 进程中断导致任务丢失 | 高 | SQLite事务保证，checkpoint机制 |
| 长时间任务（>2小时）中断 | 中 | Phase 1配置化超时，Phase 2断点续跑 |
| 多人并发使用同一数据库 | 中 | Phase 3多租户隔离设计 |

### 7.4 安全风险

| 风险 | 缓解策略 |
|------|----------|
| subprocess注入（当前P0） | 立即修复，shell=False |
| goal内容泄露敏感信息 | Worker隔离网络，敏感信息脱敏 |
| hermes chat未授权调用 | API Key管理，调用审计 |
