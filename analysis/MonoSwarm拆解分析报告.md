# MonoSwarm 系统拆解分析报告

**面向对象**：CEO 决策者  
**分析日期**：2026-05-12  
**报告性质**：技术深度评估 · 战略建议

---

## ① 执行摘要

MonoSwarm 是一个轻量级 AI 任务调度框架，核心解决了"任务拆解 → 并发执行 → 结果汇总"的三步流程，已具备可用原型。但当前版本工程化程度较低，存在任务拆解质量差（纯规则匹配）、无真正并发感知（subprocess 同步阻塞）、无监控/测试覆盖三大高优先级缺陷，短期内可通过可观测性建设和调度层改造显著提升生产可用性，长期需布局分布式架构才能支撑规模化。

---

## ② 架构总览

```
用户输入（CLI / API）
        │
        ▼
┌─────────────────────────────────────────────┐
│              CEO Brain                       │
│  职责：任务理解 → 拆解子任务 → 结果汇总        │
│  文件：ceo_brain.py                          │
│  现状：规则匹配拆解（TODO: LLM智能拆解）       │
└──────────────────┬──────────────────────────┘
                   │
        ┌──────────┼──────────┐
        ▼          ▼          ▼
   ┌────────┐ ┌────────┐ ┌────────┐
   │  code  │ │  ppt   │ │ generic│
   │ worker │ │ worker │ │ worker │
   └────────┘ └────────┘ └────────┘
        │          │          │
        └──────────┼──────────┘
                   ▼
┌─────────────────────────────────────────────┐
│         Harness（调度层核心）               │
│  职责：分发/执行/重试/汇总                   │
│  文件：harness.py                           │
│  并发：ThreadPoolExecutor（默认3并发）       │
│  重试：按max_retries次数，无指数退避         │
└──────────────────┬──────────────────────────┘
                   │
        ┌──────────┼──────────┐
        ▼          ▼          ▼
┌────────────┐ ┌────────────┐ ┌────────────┐
│WorkerPool   │ │SessionStore│ │  run.py    │
│子进程管理   │ │ SQLite持久化│ │  CLI入口   │
│spawn/监控   │ │任务/Worker │ │status/log  │
└────────────┘ └────────────┘ └────────────┘
```

**核心数据流**：用户 → CEO Brain（拆解） → Harness（调度） → WorkerPool（执行，subprocess） → SQLite（持久化） → CEO Brain（汇总） → 用户

**技术栈**：纯 Python 标准库 + 外部 Hermes CLI，无第三方依赖

## ③ 不足清单（按优先级排序）

### P0 · 必须立即处理（影响生产可用性）

| # | 不足 | 影响 | 涉及模块 |
|---|------|------|----------|
| 1 | **无单元测试 / 集成测试** | 代码迭代无安全网，改错全靠手动自测，高迭代风险 | run.py |
| 2 | **subprocess 同步阻塞** | 父进程等待所有 Worker 完成，无法真正并行；扩展性差 | harness.py / worker_pool.py |
| 3 | **任务拆解纯规则匹配** | 误判率高（如"做PPT"被错误路由），拆解质量不可控 | ceo_brain.py |

### P1 · 影响效率（建议 1 个月内处理）

| # | 不足 | 影响 | 涉及模块 |
|---|------|------|----------|
| 4 | **无真实 DAG 调度** | 并行模式是"全部并发等待"，而非按拓扑序依赖执行 | harness.py |
| 5 | **2小时超时硬编码** | 不同类型 Worker 无法差异化设置，长任务强制中断 | worker_pool.py |
| 6 | **Worker 间无数据共享** | 并行执行价值大幅缩水，上游结果无法传递给下游 | harness.py |
| 7 | **无实时日志 / 流式输出** | 无法追踪长时任务进度，故障排查困难 | worker_pool.py |
| 8 | **重试无差异化策略** | 网络超时和代码错误同等对待，无指数退避，浪费资源 | harness.py |

### P2 · 长期演进（3个月+）

| # | 不足 | 影响 | 涉及模块 |
|---|------|------|----------|
| 9 | **SQLite 单点存储** | 高并发写入可能锁表；无主从复制，无灾备 | session_store.py |
| 10 | **无分布式支持** | Worker 无法跨机器水平扩展，受单机资源上限约束 | worker_pool.py |
| 11 | **无监控/告警** | 任务失败/超时被动发现，无 SLA 指标 | — |
| 12 | **skill 版本无锁定** | 同一 Worker 类型每次可能加载不同版本 skill，行为不一致 | ceo_brain.py |
| 13 | **无多租户隔离** | 无法为不同用户/项目分配独立资源池 | harness.py / worker_pool.py |

## ⑤ 结论与建议

### 核心判断

MonoSwarm 目前处于 **"可用原型 → 生产级系统"** 的关键跃迁阶段。框架核心设计（CEO Brain → Harness → WorkerPool → SQLite）逻辑清晰，架构选型（纯 Python stdlib + Hermes CLI）符合早期轻量化原则。但工程化缺口（测试、监控、错误处理）已显著制约迭代速度和稳定性。

### 战略建议

**① 短期（1-2周）：止血优先**  
- 立即补充单元测试框架（pytest），目标覆盖率 > 60%
- 接入 prometheus_client 暴露核心 metrics，建立基础监控
- 实现日志分级（INFO/WARN/ERROR）和 worker 进度上报协议
- **优先级：最高。这三项不做，其他优化都是盲人摸象**

**② 中期（1个月）：调度能力补课**  
- 实现 worker 级超时差异化配置（code_worker 2h vs qa_worker 30min）
- 引入 DAG 拓扑排序调度（按依赖关系执行，而非"全部并发等待"）
- 统一 SKILL_ROUTING + WORKER_TYPE_SKILLS 为单一 YAML 配置源
- 实现 LLM 智能拆解替换 rule-based（调用外部 Hermes LLM）

**③ 长期（3个月+）：分布式架构**  
- 迁移 SQLite → PostgreSQL，支持高并发
- 引入 Redis/RabbitMQ 消息队列，支持 worker 分布式部署
- Docker 容器化 + K8s 调度，实现弹性扩缩容
- 构建 Plugin 系统和 skill 版本锁定机制

**不建议做的事**：短期内不要投入多租户隔离、商业化计费等高级特性——这些问题在可观测性和调度能力尚未就绪时，优化收益为零。
